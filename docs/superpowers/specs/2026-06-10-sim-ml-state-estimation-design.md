# Sim ML State Estimation — Design (Sub-project C, slice C0+C1)

**Date:** 2026-06-10 (finalized after deep research 2026-06-11)
**Status:** Finalized design, pre-implementation
**Scope:** Sub-project C first slice only — **C0** (training corpus + data foundation) and **C1** (offline state-estimation models). Builds on Sub-project A (digital twin, merged) and Sub-project B (data factory, merged). Service/Pi deployment of the models (C1-deploy) and any control/forecasting families are separate follow-on cycles.

## Research finalization — what changed

This spec was hardened by an 8-dimension literature/web review (feature engineering, grouped CV, HistGBT config, ordinal stage, sim-to-real transfer, agronomy, evaluation metrics, packaging). The load-bearing changes:

- **Biomass is DRY mass, ~0.05–5 g** (`crops/lettuce.yaml: biomass_max_g=5.0` "~150 g fresh ~5 g dry"; typical 35-day final ~1 g). The old gate band ("0–15 g", MAE ≤ 1.5 g) was larger than the signal itself — re-anchored below.
- **Sim `stage` is a pure clock** (`hal/sim/plant.py::PlantModel.stage()` is a step function of `days_elapsed` only). Feeding `days_since_start` makes stage a trivial lookup, so the old macro-F1 gate proved nothing. Now gated via a **sensors-only** ablation against a **time-only baseline**.
- **Early-stopping leakage**: HistGBT's default internal validation split would breach the by-grow firewall — fixed by passing the explicit val fold.
- **Metrics aggregate per-grow** (not over 1.5 M autocorrelated rows); **MAPE dropped** (blows up near zero biomass); **ordinal metrics added** for stage.
- **Agronomic cumulative features added** (thermal time / DLI / EC-drawdown), the literature's strongest biomass/stage drivers and the features that diverge from the clock under faults.
- **Transfer-readiness** added cheaply: perturbation robustness test, sensor-vs-time ablation, LOSO report, sim-to-real risk register.

## Goal

Train models that estimate a lettuce grow's **hidden plant state** — `biomass_g` (dry mass per plant), `health`, and growth `stage` — from the **observed sensor streams** a real farm can see (`ph_obs`, `ec_obs`, `tds_obs`, `temp_obs`) plus observable time/derived features. The digital twin gives us the hidden truth as labels; a model that beats a **time-only baseline** (especially under fault scenarios) proves the *sensor* streams carry recoverable state — the precondition for transfer to a real farm. Offline only: corpus → features → train → evaluate on held-out grows → save deployable bundle. CPU-only, no GPU.

## Key decisions

- **Deployment model:** train offline on the dev machine, export a lean joblib artifact to the Pi later. This slice stops at the saved artifact; no service wiring.
- **Input representation:** multi-scale trailing-window features (level / trend / dispersion of each observed sensor over [4 h, 1 d, 7 d] windows) plus causal **cumulative** features (thermal time, photoperiod/DLI proxy, EC-drawdown) and observable time (`days_since_start`, photoperiod phase). Backward-only (no future peeking); warmup rows dropped so no imputation.
- **Models:** scikit-learn `HistGradientBoosting` gradient-boosted trees, one independent estimator per target, each with **two** baselines it must beat: a dummy floor and a **time-only** model.
- **Targets:** `HistGradientBoostingRegressor` for `biomass_g` (`loss='absolute_error'`, monotonic in time/cumulative features), one for `health` (clipped to [0,1]), `HistGradientBoostingClassifier` for the 4-class ordinal `stage` (`class_weight='balanced'`).
- **Split:** by whole grow (never by row), scenario-stratified, via sklearn `StratifiedGroupKFold` (K=5) as the headline estimate + an optional locked test carve-out via `GroupShuffleSplit`.
- **Corpus:** ~300 grows (50 seeds × 6 scenarios) at 10-min sampling (~5040 rows/grow, ~1.5M rows), generated by B's factory from a tracked `runs/corpus.yaml`. Comfortably in-RAM for sklearn.
- **Boundary:** `ml/` depends on B only to *generate* the corpus and on B's parquet schema (`SCHEMA_VERSION="1.0"`) to *read* it. Never imports the live service or HAL drivers. New dependency: `scikit-learn` (+ `joblib`).

## Architecture & components

New top-level package `ml/` (sibling of `hal/`, `service/`, `web/`):

```
ml/
  data/
    corpus.py      # ensure_corpus(): generate via B's BatchConfig or reuse existing
    loading.py     # load_corpus(): index.json -> list[Grow]{run_id,scenario,seed,df}
    features.py    # multi-scale window + cumulative feature matrix X + aligned labels
    splits.py      # by-grow, scenario-stratified CV folds (sklearn StratifiedGroupKFold)
  models/
    estimators.py  # the 3 HistGradientBoosting models + time-only & dummy baselines
    train.py       # train_all(): corpus -> features -> CV -> fit -> evaluate -> save
    evaluate.py    # per-grow metrics, ablation, robustness, LOSO + acceptance gate
  cli.py           # python -m ml {generate-corpus, train, evaluate}
runs/
  corpus.yaml      # the ~300-grow BatchConfig (tracked; corpus output is gitignored)
artifacts/         # saved bundles (gitignored; manifest documents provenance)
```

**Data flow:** `cli` resolves `created_at`/`git_commit` + pins `OMP_NUM_THREADS` at the boundary → `corpus.ensure_corpus()` runs B's `run_batch` (or reuses a complete `index.json`) → `loading.load_corpus()` reads each run's parquet keyed by grow → `features.build_features()` produces multi-scale window + cumulative features + `y_biomass`/`y_health`/`y_stage` tagged with `run_id`+`scenario` → `splits` yields scenario-stratified grouped CV folds → `train.train_all()` fits each target's GBT + baselines per fold (early-stop on a grouped inner val) → `evaluate` scores held-out grows per-grow, runs ablation/robustness/LOSO, renders the gate → artifacts saved as a versioned joblib bundle.

**Boundary with B:** `ml/` calls `hal.sim.factory`'s `BatchConfig`/`run_batch` to make the corpus and reads its parquet/`index.json`. It validates `schema_version == "1.0"` and `index["failed"] == 0` before training so it never silently trains on a partial or stale corpus.

## C0 — corpus generation

`runs/corpus.yaml`, a `BatchConfig` for B's factory:

```yaml
name: corpus
base:
  profile: profiles/bench-sim.yaml
  duration_days: 35          # full Cornell grow
  sample_interval_s: 600     # 10-min sampling -> ~5040 rows/grow
  ic_jitter: 0.12            # distinct grows per seed
seeds: { count: 50, root: 20260610 }
scenarios: [clean, clogged_pump_midgrow, drifting_ph_probe, sensor_dropouts, chaos, random]
emit_csv: false
```

50 seeds × 6 scenarios = **300 grows**, ~1.5M rows → `data/datasets/corpus/`.

`ml/data/corpus.py` exposes `ensure_corpus(root="data/datasets/corpus", *, regenerate=False, created_at, git_commit) -> Path`:
- If `index.json` exists, all runs are `status == "ok"`, and `failed == 0` → return the path (idempotent; no recompute).
- Else load the `BatchConfig` from `runs/corpus.yaml` and call `run_batch(..., workers=...)`.
- Validate manifest `schema_version == "1.0"`; raise on any incomplete/failed corpus.

Generation is a one-time few-minute CPU batch parallelized across cores. The corpus dir is gitignored; only `runs/corpus.yaml` is tracked, so the corpus is regenerable byte-for-byte (B's determinism tests cover this) from `(corpus.yaml, git_commit)`.

> **Deferred (follow-on):** corpus-side **domain randomization** of sensor-model parameters (pH offset/gain/noise σ, EC/TDS gain, temp-compensation error, drift rate) across grows would force the model onto calibration-invariant features and is the most established route to closing the reality gap. It touches B's factory (per-grow sensor-model params), so it is out of scope for this slice; partial coverage is provided by the evaluation-side perturbation test (below), and the gap is tracked in the risk register.

## C1 — data foundation

### Loading (`ml/data/loading.py`)
`load_corpus(root) -> list[Grow]`. Each `Grow` = `{run_id, scenario, seed, df}`; `df` is the run's parquet read once. The authoritative run list is `index.json` (failed/partial runs excluded).

- **Feature inputs (observed only):** `ph_obs`, `ec_obs`, `tds_obs`, `temp_obs`, plus observable time/derived features built from them.
- **Labels (hidden truth, never fed as features):** `biomass_g` (dry mass), `health`, `stage`.
- `*_true` and all other truth columns are forbidden from `X` — the leakage firewall, asserted by test.

### Windowing + features (`ml/data/features.py`)
The model predicts current state from a backward-looking feature vector at each row `t` (after a warmup of `max(window)` samples). Three feature families:

**1. Multi-scale window features** over `WINDOWS = [24, 144, 1008]` samples (4 h / 1 d / 7 d at 10-min), per observed sensor:
- **Level:** last value (once per sensor).
- **Trend:** least-squares slope per window — captures e.g. "EC drifting down for 5 days," invisible to a 4 h window.
- **Dispersion:** rolling std per window, plus **`mean_abs_change`** (robust volatility) and a **flatline/dropout rate** (fraction of NaN/repeated samples) to make the `sensor_dropouts`/stuck-probe faults explicit.

**2. Cumulative / integral features** (causal running sums from grow start; the agronomy literature's strongest biomass/stage drivers because dry-matter accumulation is the integral of conditions, not a 4 h snapshot):
- **`gdd_cum`** — thermal time: `Σ max(0, temp_obs − T_base)·dt`, `T_base = 4.5 °C` (lettuce), plus `gdd_rate`.
- **`photoperiod_hours_cum`** — cumulative light-on hours (a DLI proxy); true cumulative PAR if a light column exists.
- **`ec_drawdown_rate`** (clipped-negative EC slope) and **`ec_drawdown_cum`** — nutrient/water-uptake proxies (near-linear in biomass per WUE literature).

**3. Time features:** `days_since_start`, `light_on`/photoperiod phase. These are observable on a real farm (transplant date is known) and stay in the deployable model, but are isolated for the ablation/baseline below because in *this sim* they trivially determine stage.

Output: a tidy `X` (one row per row `t`, fully populated — first `max(WINDOWS)` rows dropped, no imputation) + aligned `y_biomass`, `y_health`, `y_stage`, tagged with `run_id`+`scenario`. Trees take raw features (HistGBT bins inputs and is scale-invariant); the linear baselines get a `StandardScaler` via a `Pipeline`.

> EWMA (recursive, O(1)-state, Pi-streamable) level/volatility features are a noted optional addition; included if cheap, not required for the gate.

### Split (`ml/data/splits.py`)
Whole-grow, scenario-stratified, built on **sklearn** `StratifiedGroupKFold(n_splits=5)` with `groups=run_id`, `y=scenario` (battle-tested; no hand-rolled RNG to re-verify). Each CV fold holds out ~60 grows (~10/scenario); within the training grows, a further grouped `GroupShuffleSplit` carves a **val** subset used **only** for early stopping. An optional single locked test set (via `GroupShuffleSplit`) can be frozen for the deployable artifact.

**Why by-grow, not temporal:** the deployment target is a *new* grow/farm, not the future of an in-progress grow — leave-whole-grow-out is the leave-one-field-out analogue and the honest transfer estimate. A within-grow `TimeSeriesSplit` is deliberately *not* used (the model estimates state for unseen plants, it does not forecast the tail of a seen one). Documented so a reviewer doesn't "optimize" it into a leaky row split.

### Invariants (each a test)
1. No `*_true`/label column ever appears in `X`.
2. No `run_id` spans two folds (asserted on the sklearn splitter output).
3. Every feature row has fully-populated windows (no warmup NaNs).
4. Each fold contains every scenario in proportion.
5. The early-stopping val subset shares no grow with its fold's training set.

## C1 — models, baselines & training

### Estimators (`ml/models/estimators.py`)
| Target | Type | Model (key config) | Baselines it must beat |
|---|---|---|---|
| `biomass_g` | regression | `HistGradientBoostingRegressor(loss='absolute_error', monotonic_cst={time+cumulative: +1})` | `DummyRegressor(mean)` **and** a **time-only** model (`days_since_start`,`gdd_cum`,… only) |
| `health` (0–1) | regression | `HistGradientBoostingRegressor`, predictions **clipped to [0,1]** (optionally logit `TransformedTargetRegressor`) | `DummyRegressor(mean)` **and** time-only |
| `stage` (4-class ordinal) | classification | `HistGradientBoostingClassifier(class_weight='balanced')` | `DummyClassifier(most_frequent)` **and** a days-binned time-only lookup (+ optional ordinal-logistic) |

- **`loss='absolute_error'`** aligns biomass training with its MAE gate and resists fault outliers.
- **Monotonic constraints** (`monotonic_cst` as a **name→±1 dict**, requires named-column `X`) encode that biomass is non-decreasing in elapsed time and the cumulative drivers — variance reduction + a physical prior that aids transfer. A test asserts predicted biomass is non-decreasing in `days_since_start` for a fixed-other-features probe. Instantaneous pH/EC/temp stay unconstrained.
- **Bounded health:** a plain squared-error regressor can predict <0 or >1; clip at predict time (documented in the inference contract so the Pi matches) — or train in logit space.
- **Stage is ordinal** (germination<seedling<vegetative<mature). HistGBT only offers `log_loss`, so optionally also fit a **regression-then-round** ordinal head (encode 0–3, `HistGradientBoostingRegressor`, tune 3 cut-points on val for QWK) and keep whichever wins on QWK. Optional per-grow **monotone post-processing** (clamp predicted stage non-decreasing) reported alongside raw.

Hyperparameters fixed in one `TrainConfig` dataclass: `max_iter=300, learning_rate=0.1, max_leaf_nodes=31`, `early_stopping=True, n_iter_no_change=10`. Tuning is out of scope.

### Training harness (`ml/models/train.py`)
`train_all(corpus_root, *, config, out_dir, created_at, git_commit) -> TrainReport`:
1. `ensure_corpus()` → `load_corpus()` → `build_features()` → grouped CV folds.
2. Persist one feature-name list (exact columns + order; required for the monotonic-constraint dict and inference).
3. Per target, per fold: fit baselines + GBT on the fold's training grows, **early-stop by passing the explicit grouped val subset** via `fit(X, y, X_val=…, y_val=…)` (sklearn ≥1.4). **Never** rely on the internal `validation_fraction` — its default `early_stopping='auto'` turns on above 10 k rows and splits by *random row*, which would mix training grows into validation and breach invariant #2.
4. Also fit the **sensors-only** variants (time features removed) for biomass and stage — the scientific "do sensors carry signal" test.
5. Aggregate metrics per-grow across folds (below), run ablation/robustness/LOSO, save the artifact bundle, return the `TrainReport`.

`stage` string labels use a persisted `LabelEncoder`. Every estimator gets `random_state` from the config seed. **Reproducibility is bit-for-bit only at a fixed thread count** — HistGBT is OpenMP-parallel and float reduction order varies with `OMP_NUM_THREADS`, so the CLI sets and the manifest records it. No wall-clock.

## C1 — evaluation & acceptance gate

### Metrics (`ml/models/evaluate.py`)
**Aggregation:** compute each metric **per grow**, then macro-average across the held-out grows of each CV fold (and overall), reporting mean ± std (and min/max). Rows within a grow are one autocorrelated trajectory — pooling ~5040 near-duplicate rows would over-weight long grows and report false precision. Effective n is ~300 grows, not ~1.5 M rows. Reported overall **and per scenario** (clean vs. fault).

| Target | Primary | Secondary |
|---|---|---|
| `biomass_g` | MAE (g) + NMAE = MAE/IQR(biomass) | RMSE, R², **per-stage MAE** (germination/seedling/vegetative/mature) |
| `health` | MAE (0–1) | RMSE, R², **binned reliability** (predicted-vs-true per decile) + signed bias |
| `stage` | **QWK** (`cohen_kappa_score(weights='quadratic')`) | **adjacent (±1) accuracy**, ordinal MAE on 0–3, exact accuracy, confusion matrix, macro-F1 |

- **No MAPE/sMAPE for biomass** — undefined/explosive when biomass ≈ 0 (germination); NMAE(IQR) is the scale-free choice.
- **QWK + adjacent accuracy** because stage is ordinal: a germination→mature miss must cost more than germination→seedling.
- **Health reliability** catches a model that scores low MAE while regressing to the mean and never flagging an unhealthy grow.
- **Per-grow final-day biomass** (predicted vs. true harvest weight) and mean trajectory error retained — the operationally meaningful view.

### Acceptance gate
The harness prints PASS/FAIL per criterion. `--strict` makes **criterion 1 only** a non-zero exit.

1. **Sensors carry signal (binding):** the full GBT must beat **both** the dummy floor **and the time-only model** by a margin — biomass/health: **≥20% NMAE reduction vs. time-only on the fault scenarios** (where the clock breaks); stage (sensors-only model): **absolute recovery — sensors-only QWK ≥ 0.90 and adjacent accuracy ≥ 0.90**. *Recalibrated after the first full-corpus run (sensors QWK 0.9984, time-only QWK 0.9997): stage is a near-deterministic clock that every model recovers near-perfectly via cumulative time-integrals (GDD etc.), so "beat the time-only clock by a margin" is structurally unsatisfiable when both are ~1.0 — the honest binding test is absolute recovery.* Grouped permutation importance (sensor group vs. time group) is emitted so the lift's source is auditable.
2. **Absolute sanity (advisory band, derived from test-set IQR, not hard-coded):** biomass **NMAE ≤ 0.15** (report MAE in g for context; typical 35-day final ~1 g dry, cap 5 g); health **MAE ≤ 0.10**; `stage_with_time` **QWK ≥ 0.98** (confirms the clock-lookup plumbing); `stage_sensors_only` advisory: QWK within `stage_qwk_margin` of the time-only baseline. The with-time stage number is a plumbing check, **not** evidence of sensor signal.
3. **Graceful degradation (flagged):** clean-scenario metrics best; fault scenarios worse but not catastrophic (flagged, not hard-failed).
4. **Transfer robustness (flagged):** re-score the trained models on the held-out grows after **inference-time perturbations to observed inputs only** — additive Gaussian noise at 1× and 2× the sim's nominal σ, a ±0.2–0.4 pH offset, a ±5–10% EC gain error, and a slow linear pH drift. Report the degradation curve per target; flag if biomass MAE more than doubles under 1× noise + small drift (a model that collapses under mild realistic perturbation will not transfer). No retraining.

If the calibrated sim makes a band trivial or impossible, recalibrate **on val / a CV inner loop**, never from test-set results (that is selection-on-test leakage); the test grows are scored once.

### Additional reports (informational, not gates)
- **Leave-one-scenario-out (LOSO):** for each fault scenario, train on the other five and test on it — the closest no-real-data proxy for deploying into an unseen regime; tells us which fault regimes are riskiest.
- **Ablation table:** full vs. sensors-only vs. time-only per target, so the sensor contribution is explicit.

### Report artifacts
`evaluate` emits `metrics.json` (every metric, per-scenario, per-target, ablation, robustness, LOSO, gate verdicts) and `report.md` (human summary + PASS/FAIL). Both deterministic given the trained models and a fixed thread count.

## Artifacts, reproducibility & CLI

### Saved bundle — `artifacts/<id>/` (default `<id> = git_commit`)
```
biomass.joblib        # fitted HistGBR (absolute_error, monotonic)
health.joblib         # fitted HistGBR (+ clip contract)
stage.joblib          # fitted HistGBClassifier (balanced) [+ ordinal head if it wins]
baselines.joblib      # dummy + time-only + linear (audit only, never shipped)
preprocessor.joblib   # feature-name list + order, WINDOWS, T_base, scaler, LabelEncoder, health-clip
metrics.json          # full metric/gate/ablation/robustness/LOSO record
report.md             # human summary
manifest.json         # provenance (below)
```
Saved with `joblib.dump(protocol=5, compress=3)`. `preprocessor.joblib` is the inference contract: build the exact same features in the same order → predict → clip health. The three deploy models + preprocessor are the lean artifact for the Pi later (C1-deploy, out of scope).

**Security:** `.joblib` files are pickle-based and execute code on load — only load bundles produced by this repo's own train run, never an untrusted artifact. A `sha256` of each `.joblib` is recorded in the manifest so a tampered/partial bundle fails before load.

**`manifest.json`** records: `schema_version`, `corpus` (root + index hash + grow count), `git_commit`, `created_at` (passed in, not wall-clocked), full `TrainConfig` (hyperparameters + seed), a **versions block** (python, scikit-learn, numpy, scipy, joblib, pyarrow), **platform** (os, arch — documents the x86-train vs ARM-infer boundary), **`OMP_NUM_THREADS`**, **sha256 of each `.joblib`** (verified before load), and the **CV fold assignment** (run_id → fold). Given manifest + corpus + a matching pinned env, training reproduces bit-for-bit at the recorded thread count. *(Deferred to a follow-on: a separate feature-name-list sha256 and a `requirements.lock` written into the bundle — the versions block already pins the inference-critical libraries.)*

A `load_bundle(path, *, strict=True)` helper compares the manifest versions block against the running interpreter and **raises (strict) or loudly warns** on mismatch, and escalates sklearn's `InconsistentVersionWarning` to an error with manifest context — because a same-process round-trip test cannot catch the cross-version drift the Pi will hit.

> **Edge-format note (forward-looking):** the deploy artifact is joblib for now; ONNX is deferred (HistGBT artifacts are tiny and onnxruntime aarch64 wheels are ~13–18 MB, so footprint never justifies it for a first slice). If/when exported, any ONNX conversion of the trees **must use `DoubleTensorType`/float64** and be gated by a numerical-equivalence test (max abs diff < 1e-6 vs. sklearn on the test grows) — float32 rounding flips tree split-thresholds and the official sklearn-onnx tutorial shows ~99% relative error otherwise. The manifest records feature dtype to keep this door open. `skops.io` (non-pickle secure load) is a noted alternative for the trust boundary, deferred.

### CLI — `ml/cli.py` / `python -m ml`
- `python -m ml generate-corpus [--regenerate]` → `ensure_corpus()`.
- `python -m ml train [--corpus PATH] [--out artifacts/] [--strict]` → `train_all`, prints report.
- `python -m ml evaluate --artifacts PATH` → verify the saved bundle (sha256 of each `.joblib` + library versions, via `load_bundle`) and re-display the CV-computed gate from `metrics.json`. *(A true no-refit re-score is ill-defined here: the gate metrics come from per-fold out-of-fold CV predictions made during training and cannot be reconstructed from the final all-data models without refitting — so the gate is computed once at train time and integrity-verified + displayed at evaluate time.)*

`created_at`/`git_commit` resolved and `OMP_NUM_THREADS` set at the CLI boundary (B's pattern), keeping the library wall-clock-free, thread-deterministic, and testable.

### New dependency
Adds `scikit-learn` (pulls in `joblib`) to backend deps. `numpy`/`pandas`/`pyarrow` already present from B. No GPU, no new system libs.

## Sim-to-real gap — known risks

The north star is transfer to a real farm. The concrete gaps and their planned handling (tracked, not assumed away):

| # | Gap | In-scope-now mitigation | Follow-on |
|---|---|---|---|
| 1 | **Sensor-noise model** — sim noise may be Gaussian/stationary; real probes drift, foul, and respond to temperature | Robustness-perturbation report (advisory `robustness_ok` flag) — shipped | Corpus-side domain randomization of noise/offset/gain |
| 2 | **Calibration/units gap** — real `ph_obs`/`ec_obs` need per-probe calibration | Preprocessor documents expected units/ranges; manifest records them | Per-probe calibration step at deploy |
| 3 | **Unmodeled real dynamics** — cultivar, light spectrum, disease absent from the ODE | LOSO report flags regime sensitivity | Expand the twin / collect real edge cases |
| 4 | **Covariate shift** detectable at deploy | — | Two-sample / domain-classifier shift check (real feature dist vs. corpus) before trusting predictions |
| 5 | **Label gap** — real biomass/health ground truth is sparse | — | A small real validation set as the first follow-on |

## Out of scope (this slice)

- Service/API/Pi wiring of the models (C1-deploy follow-on).
- Forecasting (predict future state) and control/dosing-recommendation model families.
- Hyperparameter tuning / model search.
- **Quantile prediction intervals** (P5/P95 biomass) — deferred; would add two models per target.
- **Probability calibration** of the stage classifier (`CalibratedClassifierCV`) — deferred; the gate scores argmax (macro-F1/QWK), which calibration does not change. Recipe (prefit on the val fold, isotonic) recorded for when stage probabilities surface to UI/control.
- **`skops.io`** secure persistence and **ONNX** export — deferred (joblib + version pinning + trust note suffices for a first slice).
- **Corpus-side domain randomization** of sensor parameters — deferred (touches B's factory); risk #1 above.
- Online/incremental learning; the corpus is generated once and trained offline.

**Shipped after the first slice (run when `run_eval_extras=True`, the default):** the full ablation table (full / sensors-only / time-only / dummy per target), the robustness-perturbation report (biomass MAE degradation across noise/offset/gain levels, surfaced as an advisory non-binding `robustness_ok` gate flag), and the leave-one-scenario-out (LOSO) report.

**Still deferred (follow-on):** a separate feature-name-list sha256 and an embedded `requirements.lock` in the manifest (the versions block already pins the inference-critical libraries); corpus-side domain randomization; quantile intervals; classifier calibration; ONNX/skops; Pi deployment (C1-deploy).

## Testing

TDD throughout. Coverage:
- The five leakage/window/split invariants (incl. no `*_true` in `X`, sklearn splitter group-disjointness, val/train grow-disjointness).
- `ensure_corpus` idempotency + partial-corpus rejection.
- Feature determinism; cumulative features are causal (no future leakage) and monotone non-decreasing.
- Monotonic-constraint behavior (biomass non-decreasing in `days_since_start` on a fixed-other probe).
- Early-stopping uses the explicit val fold (assert internal `validation_fraction` is never relied on).
- Metric correctness on a tiny synthetic fixture (incl. QWK, NMAE, per-grow aggregation); gate PASS/FAIL logic incl. the time-only-baseline and sensors-only paths.
- Artifact round-trip (save → load → predict identical) **and** a version-mismatch load test (doctored manifest is rejected/warned).

Fast tests use a tiny generated corpus (few seeds, 1–2 day grows), not the full 300.
